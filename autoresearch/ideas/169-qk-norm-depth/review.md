The control identity is also unstated — the taste reviewer's last line specifically called this out: *"spec should lock step-0 identity to 016's forward... the control must be 016's WIN config, not unmodded-nanogpt, so the Δ is depth-conditional-on-016 not 'QK-norm yes/no'"*. - **No `## Plan` section.** Compare with 162/165 (sibling ideas already in the queue): they have a `## Plan` block enumerating the files (`configs/llm_config.py` field + `Tiny1M3MQKNormDepthConfig` subclass, `models/layers.py` kwarg + register + forward sites, `models/llm.py` config read + thread). Without this, code-implementer has to re-derive file locations.
- **Mechanism ambiguous — must lock Option A.** `## Design sketch` offers two alternatives: *"replaces the per-head-shared `nn.RMSNorm(d_head)` with a per-block scalar parameter `self.qk_norm_scale = nn.Parameter(torch.ones(1))` **or** a per-block `nn.RMSNorm(d_head)` (one per block rather than per head)"*. These are not equivalent. **Option A** (per-block scalar *post*-RMSNorm, applied to 016's per-head RMSNorm output) preserves 016's per-head normalization structure and adds a multiplicative gain; this matches the lever's framing ("per-block learnable scale on top of 016's WIN"). **Option B** (per-block RMSNorm instead of per-head) restructures 016's parameterization — different mechanism, different math, different parameter count. Lock **Option A**: keep `self.q_norm = RMSNorm(d_k)` and `self.k_norm = RMSNorm(d_k)` per-head (existing 016 setup, lines 1039-1040), then add `self.qk_norm_scale = nn.Parameter(torch.ones(1))` and apply `q = q_norm(q) * qk_norm_scale; k = k_norm(k) * qk_norm_scale` pre-QK-matmul.
- **Step-0 identity comparison reference is misstated.** `## Design sketch` says *"logits are bit-identical to 016's step-0 forward... fp32 max-abs-diff < 1e-3 tolerance for the RMSNorm rescaling (same as 016)"*. The reference comparison should be **169 step-0 vs 016 step-0** (the chosen control), not vs unmodded. At α_l=1.0 the per-block multiplier is exactly 1.0 ⇒ the forward graph is **byte-identical** to 016's step-0, no tolerance needed for the *169→016* check. The `fp32 max-abs-diff < 1e-3` tolerance is for the *016→unmodded* RMSNorm rescaling (already accepted by 016's spec) and should be quoted as such — don't conflate the two comparisons. Update the wording.
- **Mutual-exclusion assertions not yet specified.** If a future config accidentally combines `use_qk_norm_depth` with `use_q_only_norm`/`use_k_only_norm`/`use_qk_norm_post_rope`, the forward path becomes ambiguous. Add an explicit `assert not (self.use_qk_norm_depth and self.use_q_only_norm)` (and the symmetric for K-only, and for 016's symmetric) at the top of `MultiHeadAttention.forward`, mirroring the existing `assert not (self.use_cope and self.use_qk_norm_post_rope)` at line 1926. Same scope as the reviser-note precedent.
